**Summary of Database Management System Notes**

The main topic of this page is **Database Integrity Constraints**, which are rules that ensure the accuracy and consistency of data in a database. These constraints can be classified into two main categories: **Primary Key Constraints** and **Foreign Key Constraints**.

**Primary Key Constraints**

A primary key is a unique identifier for each row in a table. It is used to distinguish one row from another and ensure that each row has a unique value. The primary key can be a single column or a combination of multiple columns. For example, in a table that stores information about students, the student ID could be the primary key.

**Foreign Key Constraints**

A foreign key is a column or set of columns in a table that references the primary key of another table. It is used to establish relationships between tables and ensure data consistency. For example, in a table that stores information about courses, the course ID could be a foreign key that references the primary key of the students table.

**Types of Integrity Constraints**

There are several types of integrity constraints, including:

* **Not Null Constraints**: These ensure that a column cannot have a null value.
* **Unique Key Constraints**: These ensure that each row in a table has a unique value for a particular column.
* **Check Constraints**: These allow you to specify a condition that must be true for a row to be inserted or updated.
* **Domain Constraints**: These restrict the values that can be entered into a column.

**Example**

Consider a table that stores information about students, with columns for student ID, name, and age. We can apply the following integrity constraints:

* **Not Null Constraint**: The student ID column cannot have a null value.
* **Unique Key Constraint**: The student ID column must have a unique value for each row.
* **Check Constraint**: The age column must be greater than or equal to 18.
* **Domain Constraint**: The student ID column must be an integer between 1 and 100.

**Importance of Domain Constraints**

Domain constraints are important because they help ensure that data is accurate and consistent. For example, if we have a table that stores information about products, we can use a domain constraint to ensure that the price column only accepts values that are greater than or equal to 0.

**Entity Integrity Constraints**

Entity integrity constraints ensure that each row in a table has a unique identifier. They are used to prevent duplicate rows from being inserted into a table. For example, in a table that stores information about customers, the customer ID could be an entity integrity constraint.

**Key Features of Entity Integrity Constraints**

The key features of entity integrity constraints are:

* **Uniqueness**: The primary key value must be unique for each row in the table.
* **No Duplicate Entries**: No duplicate entries are allowed in the primary key column.

**Conclusion**

In conclusion, database integrity constraints are an essential part of ensuring the accuracy and consistency of data in a database. By applying these constraints, we can prevent errors and ensure that our data is reliable and trustworthy. As a database administrator, it is important to understand the different types of integrity constraints and how to apply them effectively.

This page delves into the world of SQL triggers, a crucial aspect of database management. Triggers are essentially events that occur in response to specific actions within a database, such as inserting, updating, or deleting data. They enable developers to enforce business rules, maintain data consistency, and automate tasks without requiring manual intervention.

The page highlights the importance of understanding the different types of triggers, including:

* **Before**: Triggers that execute before the action occurs.
* **After**: Triggers that execute after the action occurs.
* **Instead of**: Triggers that replace the action with a different one.

The page also emphasizes the need to consider the timing and scope of triggers, as well as their potential impact on performance.

To illustrate the concept, the page provides a simple example of a trigger that checks if a customer's age is valid before inserting a new record into the database. The trigger is defined as follows:

```sql
CREATE TRIGGER chk_customer_age
BEFORE INSERT ON customers
FOR EACH ROW
BEGIN
    IF NEW.age < 18 THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'Customer age must be 18 or older';
    END IF;
END;
```

This trigger checks if the age of the new customer is less than 18 and, if so, raises an error message. The `NEW` keyword refers to the new row being inserted, and the `IF` statement checks the condition. If the condition is true, the `SIGNAL` statement raises an error.

The page also discusses the use of trigger functions, which are reusable blocks of code that can be called from multiple triggers. These functions can simplify trigger development and improve code maintainability.
